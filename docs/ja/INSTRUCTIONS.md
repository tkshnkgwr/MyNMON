# AI開発指示書 (INSTRUCTIONS.md) - MyNMON

[English](../en/INSTRUCTIONS.md) | **日本語版**

本文書は、AIエージェント（大賢者）が `MyNMON` のソースコードを変更・保守・機能追加する際に遵守すべきガイドラインおよび開発規約です。

---

## 1. 開発原則

1. **CUI 描画品質とフリッカー防止の徹底**:
   - 画面の全体クリア (`terminal::Clear(ClearType::All)`) を行わず、カーソルを `(0, 0)` に戻した上書き描画を行うこと。
   - `UntilNewLine` による行末消去、および描画完了後の `FromCursorDown` による下部消去を組み合わせ、テキストの残存を防ぐこと。
   - 画面サイズが「横80文字以上、縦20行以上」を下回る場合は、描画処理をスキップして警告を表示すること。
2. **二重起動防止**:
   - Windows環境においては、`common_lib::check_single_instance` を用いた Named Mutex による多重起動防止ロジックを必ず維持・動作させること。
3. **リリースサイズとメモリの最適化**:
   - メモリフットプリント（25MB以下）およびバイナリサイズ（350KB以下）を維持すること。
   - 機能追加時はオプショナルな Cargo Features（条件付きコンパイル `#[cfg(feature = "...")]`）の適用を検討すること。
4. **他言語開発経験者に配慮した解説方針**:
   - ユーザーは他言語での開発経験をお持ちですが、Rust固有の専門用語（所有権、トレイト、ライフタイム等）には不慣れです。修正内容の説明時は、固有表現をむやみに並べず、一般的なプログラミング概念に置き換えるか直感的な補足解説を添えて報告してください。

---

## 2. コードスタイルのガイドライン

- **フォーマット規則**: `cargo fmt` の標準スタイルに従ってください。
- **型と変数名**:
  - 構造体 / 列挙型: `PascalCase` (例: `MonitorState`)
  - 関数 / 変数 / モジュール: `snake_case` (例: `draw_cpu_total`)
  - 定数: `SCREAMING_SNAKE_CASE` (例: `MUTEX_NAME`)
- **エラー処理**:
  - パニック (`panic!`, `unwrap()`) の使用は極力回避し、適切なフォールバック処理またはエラー値を返すようにしてください。パニック発生時は `panic = 'abort'` に従い安全にプロセスを即時停止させます。

---

## 3. テストと検証方針

- 新しいロジックを追加した場合は、対応する単体テストを追加し、テストによるカバー率を維持してください。
- 変更完了時には必ず以下の標準コマンドを実行し、警告およびエラーがゼロであることを確認してください：
  - `cargo test`
  - `cargo clippy --all-targets -- -D warnings`
  - `cargo fmt --check`
  - `cargo doc --no-deps --document-private-items`

---

## 4. モジュール分割とリファクタリング方針

- **1,000 行制限ルール**:
  - 単一のプログラムソースファイル（例: `src/*.rs`）の行数が **1,000 行** を超えたあたりで、関心事・役割に応じたモジュール分割を提案・実施し、コードの可読性と保守性を維持してください。
