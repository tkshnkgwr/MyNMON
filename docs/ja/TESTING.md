# テスト方針・実行ガイド (TESTING.md) - MyNMON

[English](../en/TESTING.md) | **日本語版**

本文書は、`MyNMON` プロジェクトにおけるユニットテスト、手動動作検証、および品質検証に関する方針と実行手順をまとめたドキュメントです。

---

## 1. テスト概要

`MyNMON` は Rust で記述された超軽量の CUI システムモニターであり、高いパフォーマンスとターミナル整合性を維持するために以下の観点から検証を行います。

主要なテスト観点：
- **ユニットテスト**: アスキーバー生成ロジックなどのアルゴリズム検証。
- **画面トグルと状態制御**: 各キー入力によるセクション表示切り替えの正確性。
- **CUI 整合性とフリッカー**: 異なるターミナルサイズやリフレッシュ頻度でのちらつき抑制や画面崩れ防止。
- **多重起動防止 (Windows専用)**: Named Mutex による二重起動検知。
- **条件付きコンパイル**: Cargo Features の有無（フル機能/最小構成等）によるコンパイル整合性。

---

## 2. テストの実行手順

ローカル環境でのテスト実行手順は以下の通りです。

### ユニットテストの実行
```bash
cargo test
```

### 特定のテストケースのみ実行
```bash
cargo test test_get_ascii_bar
```

### 品質・コード検証コマンド
コード変更完了時には、以下の標準コマンドが全てエラー・警告なしで合格することを確認します：

```bash
# 1. ユニットテスト実行
cargo test --all-features

# 2. 静的解析 (Clippy)
cargo clippy --all-targets -- -D warnings

# 3. コードフォーマット検証
cargo fmt --check

# 4. Rustdoc ビルド検証
cargo doc --no-deps --document-private-items
```

---

## 3. テストの記述ガイドライン

1. **モジュール内テスト**:
   - 各モジュールの末尾（例: `src/utils.rs` 内）に `#[cfg(test)] mod tests` を配置し、アサーションロジックをテストします。
2. **安全なエラーハンドリングのテスト**:
   - 異常な環境や値が入力された場合でもプログラムがパニックせず、適切なデフォルト値またはエラーを返すことを保証します。
3. **Features による条件分岐の配慮**:
   - `#[cfg(feature = "...")]` に関連するロジックやテストは、対応する Feature フラグが指定されていない場合でもコンパイル可能である必要があります。
