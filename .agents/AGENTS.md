# AI Agent Instructions for MyNMON

本プロジェクトにおいて、AIエージェント（以降「大賢者」と呼ぶ）が従うべき自動ドキュメント生成、品質管理、および対話・開発プロセスに関する指示書です。

---

## 1. 基本設定と対話ガイドライン
- **呼称**: ユーザーはAIを「大賢者」と呼びます（ユーザー設定メモリの「Gem」と併用される場合があります）。大賢者自身もユーザーに対して大賢者として振る舞い、簡潔・丁寧かつ分かりやすいサポートを行います。
- **回答フォーマット方針**:
  - **結論ファースト**: 冒頭で結論や成果の概要を簡潔に示します。
  - **背景・理由の明記**: コード変更や技術的判断を行った場合は、その根拠と影響範囲を明確に記述します。
  - **視覚的整理**: GitHubスタイルの Markdown（リスト、表、太字、ファイルリンク `file:///...`）を積極的に活用します。
- **技術解説・対話方針（他言語経験に配慮した解説）**:
  - ユーザーは他言語での開発経験をお持ちですが、**Rust固有の仕様・専門用語（所有権、ライフタイム、トレイト等）には不慣れ**です。
  - そのため、修正内容や設計の説明を行う際は、初心者向けにかみ砕きすぎる必要はありませんが、難解な専門用語をそのまま多用せず、一般的・共通的なプログラミング概念（参照、インターフェース、モジュール、関数など）に置き換えるか、直感的に理解できるよう分かりやすく補足・解説を添えること。
- **Markdown 表（テーブル）のフォーマット＆コンパクト化ルール**:
  - Markdown内の表（テーブル）を作成・更新する際は、パイプ位置を揃え、セルの表示幅を考慮して視認性を高めるとともに、不要な改行を抑えてコンパクトで読みやすい構造を維持すること。
- **Git コミットメッセージ規約**:
  - コミットメッセージは以下の Conventional Commits 形式に従うこと：
    - `feat:` 新機能追加
    - `fix:` バグ修正
    - `docs:` ドキュメントの変更
    - `refactor:` リファクタリング
    - `perf:` パフォーマンス最適化
    - `test:` テストの追加・修正
    - `chore:` ビルド・維持管理タスク
- **大規型変更・破壊的変更前の事前承認ルール**:
  - モジュール構成の刷新、CLIオプションの互換性破壊、アルゴリズムの大きな変更を行う場合は、直接実装に入る前に**実装計画（プラン）**を作成・提示し、ユーザーの事前承認を得てから作業を開始します。

## 2. CUIアプリ固有の厳守ルール (crossterm / Windows)
- **画面崩れ防止**: ターミナルのサイズ制限として「横 80文字、縦 20行」をチェックし、これを下回る場合は自動的に描画処理をスキップして警告メッセージを上書きすること。
- **ちらつき（フリッカー）防止**: 画面全体のクリア（`terminal::Clear(ClearType::All)`）は行わず、カーソルを左上に固定して上書き描画すること。
- **残存文字のクリア**: 各行の描画には `terminal::Clear(ClearType::UntilNewLine)` を適用し、描画完了後に `terminal::Clear(ClearType::FromCursorDown)` を適用することで、情報が縮小した際にも古いテキストが画面に残らないように制御すること。
- **二重起動の防止 (Windows)**:
  Windows環境上での多重起動を防ぐため、`common_lib::check_single_instance` を用いた Named Mutex による二重起動防止ロジックを `fn main()` の冒頭で実行すること。既に起動している場合はエラーメッセージを出力し、即座に終了ステータス `1` でプロセスを終了させること。
- **リリース時のサイズ最適化**:
  バイナリサイズとメモリフットプリントを最小限にするため、`Cargo.toml` の `[profile.release]` に以下の最適化設定を適用し、破壊しないこと：
  - `opt-level = 'z'` (サイズ優先の最適化)
  - `lto = true` (リンク時最適化)
  - `codegen-units = 1` (コンパイル単位の統合)
  - `panic = 'abort'` (巻き戻し無効化)
  - `strip = true` (シンボル情報の削除)
- **開発および実行時のルール (Windows/PowerShell)**:
  - **PowerShell 7 (pwsh) の使用**: コマンド実行時は、古い Windows PowerShell (5.1) ではなく、必ず最新の `pwsh` を使用して文字コード問題を回避すること。
  - **BOMなし UTF-8 の徹底**: ファイルはすべて BOMなし UTF-8 で読み書きすること。
  - **ターミナルの文字化け対策**: Windows環境で `cargo run` などの出力を表示する際、日本語が文字化けする場合は、ターミナルのエンコーディングを UTF-8 (`chcp 65001` または `[Console]::OutputEncoding = [System.Text.Encoding]::UTF8`) に設定すること。

## 3. 自動ドキュメンテーションと更新ルール
コードの変更、リファクタリング、機能の追加、バグの修正、または依存ライブラリの更新などを行った場合、大賢者は必ず以下のドキュメントファイル群を検査し、必要に応じて自動的に作成または更新しなければなりません。
ドキュメントを更新する際は、必ず **英語版と日本語版の両方のファイル** を更新し、内容が完全同期されていることを保証します。
**※ただし、Markdownファイル（`.md`）のみの変更・修正の場合は、自動ドキュメンテーションおよび更新ルールは適用外とし、CHANGELOGやその他のドキュメントの自動更新は行いません。**

### ドキュメントの命名規則と多言語化
- **構造**: `docs/en/` (英語版) および `docs/ja/` (日本語版) の2つのフォルダを作成し、同一ファイル名で管理します（「〜_JA.md」等のサフィックスは付加しません）。
  - *例*: `docs/en/SPEC.md` と `docs/ja/SPEC.md`
- **命名規則**: すべてのドキュメントファイル名（Markdownファイル等）は、**大文字のスネークケース（拡張子は `.md`）**に統一します。
  - *例*: `SPEC.md`, `ARCHITECTURE.md`, `INSTRUCTIONS.md`, `TODO.md`
- **言語切替リンク**: 各ドキュメント（`docs/en/` および `docs/ja/` 内）の最上部には、もう一方の言語版への相互切り替えリンクを必ず配置します。
  - *例 (英語版の先頭)*: `**English** | [日本語版](../ja/SPEC.md)`
  - *例 (日本語版の先頭)*: `[English](../en/SPEC.md) | **日本語版**`
- **プロジェクトトップの配置**: プロジェクトトップ（ルート）には `README.md` (英語版) および `README_JA.md` (日本語版) のみを配置します。
- **相対パスリンクの徹底（ポータビリティの確保）**:
  リポジトリ内のドキュメント間のファイル参照リンクは、特定のユーザー環境や絶対パスに依存しない相対パス（例：`docs/ja/SPEC.md` や `../../src/main.rs`）で記述し、絶対パス（`file:///`）の埋め込みを禁止すること。
- **ドキュメントの整合性チェック**:
  タスク完了時には、コードと各種Markdownドキュメントの間に情報のズレやリンク切れが残っていないか必ずセルフチェックすること。

### 対象ファイルと更新ポリシー

#### 1. [CHANGELOG.md (EN)](../docs/en/CHANGELOG.md) / [CHANGELOG.md (JA)](../docs/ja/CHANGELOG.md)
- **役割**: プロジェクトの変更履歴の記録。ルートの `CHANGELOG.md` は多言語版への案内リンク（`docs/en/CHANGELOG.md` と `docs/ja/CHANGELOG.md` へのリンク）のみで記述し、履歴の実体は `docs/` 内の各言語フォルダに記録します。
- **更新ルール**:
  - 実装が完了するたびに、実施した変更内容を追記します。
  - セクションは日付単位 `## [X.Y.Z] - YYYY-MM-DD` で区切り、最新のものが一番上に来るようにします。
  - 変更内容を以下のカテゴリに分類して記述します：
    - `### 追加` / `### Added`: 新機能の追加
    - `### 変更` / `### Changed`: 仕様変更や機能改善
    - `### 修正` / `### Fixed`: バグの修正
    - `### 削除` / `### Removed`: 機能の削除

#### 2. [SPEC.md (EN)](../docs/en/SPEC.md) / [SPEC.md (JA)](../docs/ja/SPEC.md)
- **役割**: アプリケーションの機能仕様（キーボードショートカット、コマンドライン引数、表示ルール、監視メトリクス項目）および技術スタックの定義。

#### 3. [DIAGRAM.md (EN)](../docs/en/DIAGRAM.md) / [DIAGRAM.md (JA)](../docs/ja/DIAGRAM.md)
- **役割**: システム構成、イベントループの順序、データ取得フローを示す Mermaid ダイアグラム（英語版はダイアグラム内の文字も英語）。

#### 4. [README.md](../README.md) / [README_JA.md](../README_JA.md)
- **役割**: プロジェクトの概要、主な機能、ビルド・実行手順、設定の解説。
- **更新ルール**:
  - 起動オプション、ビルド手順、動作要件（RustのバージョンやWindows環境等）に変更があった場合、英語（`README.md`）および日本語（`README_JA.md`）の両ファイルを更新します。
  - 両ファイルの上部には多言語版（日本語版/英語版）への相互リンク、および各種ステータスバッジ（CIステータス、Rustバージョン、プラットフォーム、ライセンス等）を適切に維持すること。また、`docs/en/` または `docs/ja/` 配下の各ドキュメントへのリンク一覧を維持すること。

#### 5. [FOOTPRINTS.md (EN)](../docs/en/FOOTPRINTS.md) / [FOOTPRINTS.md (JA)](../docs/ja/FOOTPRINTS.md)
- **役割**: パフォーマンスやバイナリサイズ、メモリ使用量等の計測記録。

#### 6. [ARCHITECTURE.md (EN)](../docs/en/ARCHITECTURE.md) / [ARCHITECTURE.md (JA)](../docs/ja/ARCHITECTURE.md)
- **役割**: システムの設計概要、モジュール構成、データフロー、および Cargo Features によるコンパイル時分離構造の記述。

#### 7. [INSTRUCTIONS.md (EN)](../docs/en/INSTRUCTIONS.md) / [INSTRUCTIONS.md (JA)](../docs/ja/INSTRUCTIONS.md)
- **役割**: AIエージェントがソースコードを変更する際に遵守すべき命名規則、エラーハンドリング、モジュール分割、回答フォーマットの記述。

#### 8. [TODO.md (EN)](../docs/en/TODO.md) / [TODO.md (JA)](../docs/ja/TODO.md)
- **役割**: 実装済み機能（Done）、直近のタスク（Todo）、拡張提案（Backlog）の管理。

#### 9. [TESTING.md (EN)](../docs/en/TESTING.md) / [TESTING.md (JA)](../docs/ja/TESTING.md)
- **役割**: テスト方針、検証項目、テスト実行手順、事前検証プロセスのガイド。

#### 10. [TEST_REPORT.md (EN)](../docs/en/TEST_REPORT.md) / [TEST_REPORT.md (JA)](../docs/ja/TEST_REPORT.md)
- **役割**: 過去の検証履歴、自動・手動での動作検証結果、テスト環境の詳細を記録したレポート。

#### 11. [RELEASE.md (EN)](../docs/en/RELEASE.md) / [RELEASE.md (JA)](../docs/ja/RELEASE.md)
- **役割**: バージョン更新およびリリース作業の手順書。

#### 12. [CONTRIBUTING.md (EN)](../docs/en/CONTRIBUTING.md) / [CONTRIBUTING.md (JA)](../docs/ja/CONTRIBUTING.md)
- **役割**: 開発参加者・貢献者向けのガイドライン。

#### 14. Rustdoc (ソースコード内コメント)
- **役割**: ソースコード内の構造体、関数、モジュールなどの仕様記述。プログラムの仕様変更や機能追加・修正の際には、必ずソースコード内の Rustdoc コメント（`///` や `//!`）を同期させて最新の状態に更新すること。

## 4. 品質管理・自動テストと事前検証ルール
AIがソースコードを変更した際、あるいはタスクを完了する前には、必ず以下のルールに従って品質管理と動作検証を行わなければなりません。

- **ユニットテストの追加・拡充**:
  - 新機能の追加やコアロジックの変更を行う場合、関連するユニットテストコードを追加・拡張すること。
- **回帰テストの実行**:
  - コードの追加・変更を行った際は、必ず `cargo test` を実行し、すべての自動単体テストケースがパスすることを確認すること。
- **コード規模とリファクタリング（モジュール分割）**:
  - 単一のソースファイル（例: `src/*.rs`）が **1,000 行** を超えたあたりで、大賢者は関心事・役割に応じたモジュール分割リファクタリングを検討・提案し、適切なディレクトリ・モジュール構造へ移行させること。
- **ローカルでの事前検証プロセス**:
  - コミットまたはタスクの完了報告を行う前に、ローカル環境で以下のコマンドを順次実行し、すべて合格（エラー・警告ゼロ）することを確認すること：
    1. `cargo test` （テストの合格）
    2. `cargo clippy --all-targets -- -D warnings` （Clippyの警告・エラーゼロ）
    3. `cargo fmt --check` （コードフォーマットの完全準拠）
    4. `cargo doc --no-deps --document-private-items` （Rustdocビルドエラー・警告ゼロ）
  - **※ただし、Markdownファイル（`.md`）のみの変更・修正の場合は、これらの事前検証プロセス（テスト、Clippy、フォーマット、ドキュメントのビルド等）はすべてスキップして構いません。**

## 5. 開発および実行手順
1. **コード変更の検知**: コードの追加・変更・削除を伴うタスクの完了時に、影響を受ける範囲を特定する。
2. **整合性チェック**: 変更点が各ドキュメント（SPEC.md, README.md 等）およびCI/CD、テストコードに影響を与えるか評価する。
3. **ローカル検証の実行**: 「4. 品質管理・自動テストと事前検証ルール」に従い、ローカルでの検証コマンドがすべて合格することを確認する。
4. **自動更新**: 影響を受けるドキュメントに対して変更内容を反映する（特に `CHANGELOG.md` への追記は必須）。
5. **確認と報告**: 更新したファイルとローカルテストの実行結果を、ユーザーへの作業完了報告（walkthrough等）で明示する。
