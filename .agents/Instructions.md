# AIエージェント向け指示書 (Instructions.md)

本ドキュメントは、AIエージェントが `MyNMON` プロジェクトのコードを修正・追加する際に遵守すべき開発の原則、コーディング規約、および振る舞いについての制約を定義したものです。

---

## 1. 命名規則とコードスタイル

Rust の標準的な命名規則（API Guidelines）を厳守してください。

- **変数・関数・モジュール名**: `snake_case` (小文字とアンダースコア)
- **構造体・列挙型・トレイト名**: `UpperCamelCase` (キャメルケース)
- **定数・スタティック変数**: `SCREAMING_SNAKE_CASE` (大文字とアンダースコア)
- **ファイル名**: `snake_case`
- **インデント**: スペース4つ。改行コードは `LF` を使用してください（`.editorconfig` に準拠）。
- **ドキュメンテーション**:
  - 新規の構造体、関数、公開 API には、必ず Rustdoc 形式（`///` または `//!`）のドキュメントを記述してください。
  - プロジェクト全体のローカライズ方針に従い、インラインコメント、ブロックコメント、および Rustdoc はすべて **日本語** で記述してください。

---

## 2. エラーハンドリングの方針

クラッシュや想定外のパニックを防ぐため、安全で明示的なエラーハンドリングを行ってください。

- **例外の回避**:
  - `unwrap()` や `expect()` の使用は避けてください（単体テスト内を除く）。
  - エラーの可能性がある操作は、`Result<T, E>` または `Option<T>` を戻り値として返してください。
- **エラーの伝播と復元**:
  - `crossterm` や `sysinfo` の操作中に発生したエラーは `?` 演算子を用いて呼び出し元に伝播させ、`main.rs` の最上位で適切にキャッチしてターミナルの復元処理（生モードの解除等）を実行した上で終了してください。
- **フォールバック処理**:
  - ハードウェア情報（ディスクのマウントポイント、ネットワークインターフェース名など）が取得できない場合は、処理をパニックさせず、`"Unknown"` などのデフォルト値や、空のデータでフォールバックして実行を維持してください。

---

## 3. コンポーネントおよびモジュールの分割基準

コードの役割を明確に分担し、スパゲッティコード化を防いでください。

- **関心の分離 (Separation of Concerns)**:
  - 状態管理は [state.rs](../src/state.rs) (例: `MonitorState`) に集約してください。
  - レンダリング（画面出力）ロジックは [ui.rs](../src/ui.rs) に記述し、ビジネスロジックやメトリクス収集と分離してください。
  - 単位変換、文字列のパディング、ASCII進行バー生成などの計算系ヘルパーは [utils.rs](../src/utils.rs) に記述し、単体テストを配置してください。
- **共通ロジックの外部化**:
  - 複数のプロジェクト（例: デスクトップアプリ等）で使い回せるような汎用的ユーティリティ（二重起動防止やテキスト差分など）は、本体の `src/` に直書きせず、共有ライブラリである `common_lib` に追加・実装し、それを呼び出すようにしてください。

---

## 4. AI 出力フォーマットの指定

効率的で誤解のないコード生成のために、AI は以下のフォーマットと対話スタイルを遵守してください。

- **解説の最小化**:
  - コード変更の提案を行う際、コード自体の背後にある設計意図や変更の要点のみを簡潔に説明し、冗長なソースコード解説や単なるコードの再掲は最小限に抑えてください。
- **正確な部分置換 (diff / replacement chunks)**:
  - ファイルの全書き換えは極めて高コストであり、バグを誘発しやすいため、変更が必要な一部分のみを対象とした正確な置換指示（`replace_file_content` ツールなど）を使用してください。
- **ドキュメントの自動更新義務**:
  - コードを変更した場合は、必ずプロジェクトの自動更新ルール（`.agents/AGENTS.md`）に従い、以下のドキュメントを更新してください。
    - [CHANGELOG.md](../CHANGELOG.md): 変更履歴の自動追記
    - [docs/SPEC.md](../docs/SPEC.md) / [docs/DIAGRAM.md](../docs/DIAGRAM.md): 仕様・設計図への反映
    - [docs/Todo.md](../docs/Todo.md): タスク完了マークと新規タスクの追加
- **ローカル検証の必須化**:
  - コードの修正を完了する前に、必ず `cargo clippy` と `cargo test` をローカル環境で実行し、エラーや警告が一切ないことを確認してから完了報告を行ってください。
